前言

​ 之前使用zerotier延迟非常低，但是最近两天经过校园网整改后，有人就跟我反映之前那台服务器已经无法连接了，而那台服务器就是没有登录校园网账号，完全使用ipv6打洞成功，猜测校园网的ipv6已经无法打洞了，随后在无法连接的服务器登录校园网，发现可以秒连接上，但是使用起来却非常卡，经过ping后发现延迟非常高，且及其不稳定，这时才恍然大悟：以后都是走的ipv6打洞成功且延迟非常低，封禁之后，ipv4的打洞成功率非常低，因此都是使用中继传输的方式

​ 这时候，不得不自己搭建一个中继服务器以降低延迟，提高使用体验，加之之前本来想尝试连接局域网设备的玩法（通过该子网的某台使用zerotier或tailscale的设备连接该子网下的所有设备），但是zerotier添加路由竟然非常收费，因此这里就开启tailscale的坑

但是这里为什么非要做这样一个操作呢？

查询了下资料，hi.ServerName主要是客户端请求tls握手时携带的访问的主机名，这要是基于SNI的扩展

来自于chatgpt的回答

因为一个 Web 服务器通常负责多个主机名 - 或域名（它们是网站的人类可读名称）。如果网站使用 HTTPS，则每个主机名都将拥有自己的 SSL 证书，而这时客户端没有携带主机名，则服务器不知道时选择哪个证书。

在没有SNI的情况下，SSL/TLS握手阶段会向服务器发送一个“客户端Hello”消息，其中包含了对加密套件的支持、客户端支持的加密协议版本等信息，但并没有指定访问的具体域名。这意味着服务器无法知道客户端请求的是哪个网站，因此只能使用一个默认证书（或无法完成握手）。而在支持SNI的情况下，客户端在“客户端Hello”消息中会额外提供它所请求的主机名（即域名），这使得服务器可以根据该信息返回相应的证书。

具体步骤如下：

  1. 客户端发起握手请求：当客户端向服务器发起TLS握手时，它会在“客户端Hello”消息中加入一个SNI扩展字段，其中包含它所请求的域名。
  2. 服务器根据SNI选择证书：服务器接收到该信息后，根据请求的域名选择相应的SSL/TLS证书。比如，如果同一台服务器托管了多个网站，服务器能够根据SNI字段返回不同的证书，确保每个网站都能有独立的加密通道。
  3. 服务器返回证书：服务器向客户端返回选定的证书（包括公钥等信息），完成SSL/TLS握手。
  4. 加密通信开始：双方在建立加密连接后，可以继续进行安全的通信。

而上面的代码中hi.ServerName就是客户端携带的请求域名，而m.hostname则是证书的域名，在无证书使用自签的状态下，ServerName会为空白（还是使用的ip访问），而hostname则是使用的自签证书的随意域名，这个时候如果不注释这3行，就会报错。

而在这条commit中https://github.com/tailscale/tailscale/commit/87546a5edf6b6503a87eeb2d666baba57398a066

tailscale已经加入了ip作为证书的修复，如果使用ip自签名证书的条件下，则不会注释掉这3行代码也不报错，因为在上述commit中，已经进行ip自签名证书的更改。
